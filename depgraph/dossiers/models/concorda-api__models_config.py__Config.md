# Config

## Purpose

The `Config` model serves as a centralized, namespace-based key-value store for application-level settings. It stores arbitrary JSON payloads in the `data` field, allowing different parts of the system to retrieve configuration-specific parameters without needing a dedicated table for every new feature flag or setting.

## Invariants

- **`id` is a UUID string.** The primary key is a 36-character string generated by a `uuid.uuid4()` lambda.
- **`namespace` is a unique identifier.** Every configuration entry must have a unique string-based namespace to prevent collisions between different feature modules.
- **`data` is a JSON object.** The field is non-nullable and must contain a valid dictionary/JSON structure.
- **`updated_at` is an optional DateTime.** It tracks the last modification time of the configuration entry.

## Gotchas

- **Schema redesign impact.** Per commit `ee82e42`, this model was part of a recent schema redesign involving relationship tables and data migrations; any changes to the `data` JSON structure must be compatible with existing migration scripts to avoid breaking the new relationship-heavy architecture.

## Cross-cutting concerns

- **Auth**: None.
- **Websocket**: None.
- **Audit**: N/A.
- **Rate limit**: None.
- **Side effects**: Changes to specific namespaces may trigger logic in the service layer (e.g., feature flag toggles), but the model itself is a passive storage layer.
