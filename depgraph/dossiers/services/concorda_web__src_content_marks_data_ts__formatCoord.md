# formatCoord

## Purpose
Frontend helper that formats a coordinate string from the published MBSA Standard Rounding Marks list into a single human-readable string for display in the marks table. The input format is the GSI's own "DD.MM.MMMM" (degrees + decimal minutes packed into one dotted string, e.g. `"42.34.2406"` = 42Â° 34.2406'), which is awkward to read directly; this function expands it to one of three sailor-friendly renderings (`dd`, `dmm`, `dms`) and appends the hemisphere letter. It is the read-side counterpart to `toDecimalDegrees` in the same file, and it exists so the marks page can let the user toggle coordinate display style without touching the underlying authored data.

## Invariants
- Input must be the GSI-published "DD.MM[.MMMM]" form. `lat` is treated as N, `lng` as W (positive West) â€” both hemispheres are hard-coded at the call sites.
- A missing/undefined `coord` returns `""` (not `undefined`); callers rely on this to render empty cells for portable/movable marks (designators E, F, W) that have no fixed position.
- A coord with no `.` is returned verbatim â€” degenerate fallback, not a normal path.
- The three `CoordFormat` values (`dd`, `dmm`, `dms`) must stay in sync with `COORD_FORMAT_LABEL` keys and the format toggle in `marks-content.tsx`.
- Pure function, no I/O, no React, no locale-dependent calls â€” safe to call during render.

## Gotchas
- The "DD.MM.MMMM" parsing is positional on dots, not regex â€” `coord.slice(0, firstDot)` for degrees, then split the remainder on the next dot. A coord like `"42.17.8"` (designator 34/35/HB, no fractional minutes after the decimal point in the published list) is valid and parses as 42Â° 17.8'; do not assume a fixed minute precision.
- `dms` seconds are padded with `padStart(4, "0")` against `seconds.toFixed(1)` â€” that's `MM.M` width (4 chars). If you bump to `toFixed(2)` the pad width must change too, or sub-10 seconds will misalign.
- This file is the source of truth for `/public/marks.gpx`; if you change parsing semantics, regenerate via `scripts/generate-marks-gpx.ts` so the GPX export stays consistent with on-screen values.
- Only one commit has touched this file (`d647124` â€” initial yearbook content pages). No fix/revert history yet; treat parsing edge cases as un-battle-tested.

## Cross-cutting concerns
None. No auth, no network, no audit, no websocket. The function is invoked synchronously from the marks page render path (`MarkTable` rows and the `MarksContent` header sample) and has no side effects.

## External consumers
None known. The marks data is also exported as static GPX at `/public/marks.gpx`, but that file is generated by a separate script (`scripts/generate-marks-gpx.ts`) that does its own DD-to-decimal conversion and does not call `formatCoord`.

## Open questions
- Should the empty-coord case render a dash/placeholder instead of `""` so portable marks (E/F/W) read as "intentionally unlocated" rather than "data missing"? Currently the table just shows blank cells.
- Hemisphere is fixed N/W at every call site. If MBSA ever publishes marks south of the equator or east of the prime meridian (unlikely for Mass Bay, but the type allows it), nothing enforces correctness â€” the `"N" | "W"` union is a soft constraint, not a check against the value.
