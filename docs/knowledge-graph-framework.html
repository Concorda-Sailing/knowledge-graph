A Conceptual Framework for the Knowledge Graph

Three layers of facts about a software system, and the simple shape that holds them.

The starting idea

Software systems are full of relationships. A function uses another function. A page calls a server. A business rule constrains a sign-up flow. These relationships exist whether or not anyone writes them down — and they exist whether or not anyone agrees on what they should be. The system described here writes them down in a single, consistent shape called a graph.

A graph, in this document, is the simplest possible structure for "what relates to what." It has only two ingredients, defined in the next two sections.

Nodes — the things the graph knows about

A node is one identifiable thing. Something you can point to and give a name. In this system, a node could be:

  • a function inside a code file,
  • a service running on a server,
  • an endpoint reachable at a URL,
  • a domain concept like "post draft,"
  • a business rule like "posts must be reviewed before publishing."

Every node has an identity (a stable name the graph uses to refer to it) and a small set of facts about what it is — its kind, where it lives, when it was last updated. Nothing more. Nodes do not contain other nodes; they sit alongside one another, each one a peer.

Edges — the connections between nodes

An edge is a labeled connection between exactly two nodes. It says: this node relates to that node, in this specific way. An edge carries three things:

  • A type. What kind of relationship this is. "Calls." "Imports." "Publishes to." "Governs." "Owns." The type is a small fixed vocabulary; the graph only allows types it understands.
  • A direction. Which side initiated the relationship. The function that does the calling sits on one side; the function being called sits on the other. Most edges are directional in this sense. A few are symmetric, where the two nodes are peers on a shared channel and neither one started it — the edge type marks those explicitly so they aren't read as caller-and-callee by mistake.
  • Attributes. Small extras describing the edge itself: the file and line where the relationship is visible, how confident the graph is in the connection, a reference to the contract being honored, and so on.

Why nodes and edges are enough

Once a system is described as nodes and edges, you can answer questions by walking the graph: starting at one node, following its outgoing edges to the nodes they lead to, then following their outgoing edges, and so on. The shape of every question stays the same; the answers come from how the graph is filled out. Examples:

  • "What does this function rely on, directly and through the chain?" — start at the function node, walk out along edges of type calls and imports.
  • "What would break if I removed this endpoint?" — start at the endpoint node, walk inward along edges of type calls to find every place that depends on it.
  • "Which rules govern this process?" — start at the process node, walk inward along edges of type governs.

A graph is useful in proportion to how complete and how trustworthy it is. The framework is built around protecting both: extractors keep it full, and validation rules keep it honest.

Three graphs, one system

A single graph of everything would be unreadable and impossible to maintain. Different kinds of facts answer different kinds of questions, and they come from different sources of truth. The framework keeps three graphs side by side, each one focused on one kind of fact.

depgraph — the code graph

depgraph records what is actually in the source code. Its nodes are the things a programmer can point to in a file: functions, classes, modules, the long-lived variables that hold a server or a database connection. Its edges are the relationships visible in the code itself: this function calls that one (at this exact line), this file imports that module, this class is built on top of that base class.

The source of truth is the source code. When the code changes, depgraph re-reads the affected files and updates its nodes and edges. Anything depgraph claims can be checked by opening the file at the line the edge points to.

logigraph — the meaning graph

logigraph records what the system means — the concepts, rules, and processes the software is meant to embody. Its nodes are not lines of code; they are ideas: "post draft," "content category," "publish flow," "drafts must be reviewed before publishing." Its edges describe how those ideas relate: this rule governs this process, this concept belongs to that larger concept, this step comes before that step.

The source of truth here is human judgment. logigraph is written and maintained deliberately, in plain prose, and reviewed. It is the layer where intent lives — the layer where you can ask "why does this exist?" and expect an answer that isn't just a pointer to a line of code.

comgraph — the communication graph

comgraph records how the parts of the system talk to one another across boundaries. Its nodes are services (a backend server, a web client, a mobile app, an outside provider like a payments processor) and the addressable channels those services use to communicate — endpoints reachable at URLs, topics on a message bus, named files, signal handlers. Its edges are the interactions themselves: this service calls that endpoint, this publisher sends messages to that topic, this subscriber listens for them.

Communication has shape, and comgraph records it through the direction on its edges:

  • Directional — one side initiates, the other responds. A web client calling a backend endpoint. A publisher sending an event to a topic. Most interactions are this shape.
  • Bidirectional — a persistent open channel where either side can speak first. A live notifications stream between a server and a browser, for example. Rarer, but real.
  • Fan-out — one publisher sending to a channel that many subscribers listen on, with the publisher unaware of who's listening.

Notice that the endpoint and the topic in the diagram above are nodes of their own — not just labels on edges. That's deliberate. An endpoint can exist with no caller (a dead endpoint, deployed but unused). A topic can exist with no subscriber (a publisher firing into the void). If those addressable channels were collapsed into edge labels, the graph could not distinguish "unused" from "absent." Giving them their own nodes makes the orphan visible the moment it appears.

How the three layers compose

The three graphs are not isolated. A single concern — say, the act of publishing a post — appears in all three layers at once: as functions and classes in the code (depgraph), as a process and the rules governing it (logigraph), and as endpoints and messages travelling between services (comgraph). The layers join through cross-graph references: edges that point from a node in one graph to a node in another.

With cross-graph references in place, the system can answer questions no single layer could answer on its own:

  • "What code implements this business rule?" — start at the rule in logigraph, follow its governs edges to the endpoints in comgraph it constrains, then follow each endpoint's cross-graph reference into the function in depgraph that declared it.
  • "What would break if we changed this endpoint's contract?" — start at the endpoint in comgraph, walk incoming calls edges to find every service that depends on it, then follow each service's references back into depgraph to land on the exact call sites that need updating.
  • "Is this endpoint orphaned?" — find every endpoint node in comgraph with no incoming calls edges. Each one is either dead, or about to be.

What this gives us

None of these questions ask the graph to be "smart." They ask the graph to be complete and correct: full enough that the answer is in there somewhere, and trustworthy enough that you can act on what you find. The whole framework is built around protecting those two properties.

The payoff is that the system as a whole becomes legible. Anyone — a new contributor, the author six months later, a tool reasoning about a change — can walk in at any node and follow the edges out to the surrounding context, in whichever layer is relevant. The graph is the shared map.

Two of the three layers are running today: depgraph reads the source code and maintains the code-level graph; logigraph holds the deliberately-written domain and rules. comgraph is the proposed third layer — a graph of communication between services. Its shape, as described here, is the working sketch: services and addressable channels as nodes, interactions as typed directional edges, with cross-graph references binding the three layers into a single picture.
